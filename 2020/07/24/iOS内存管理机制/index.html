iOS内存管理机制

发表于 2020-07-24
字数统计:

问题的提出

当我们声明一个变量的时候，就使用了系统分配给我们的一块内存空间，但是当数据不再使用的时候，占用的内存空间该如何被释放？

内存中的五大区域

  • 栈：存储局部变量，当局部变量的作用域被执行完毕之后，这个局部变量就会被系统自动回收。
  • 堆：OC对象.使用C函数申请的空间。
  • BSS段：存储未初始化的全局变量、静态变量。一旦初始化就回收并转存到数据段之中。
  • 数据段：已经初始化的全局变量、静态变量。直到程序结束的时候系统会自动回收存储在代码段中的数据。

  1. 栈、BSS段、数据段、代码段存储在它们中的数据的回收,是由系统自动完成的.不需要我们干预。存储在堆中的OC对象,系统不会自动回收. 直到程序结束的时候才会被回收.

  2. 栈的空间相对于堆来说较小，但是效率高。

结构体与类的使用

  1. 结构体变量分配在栈，OC对象分配在堆。
  2. 如果需要封装的数据只有属性没有方法可使用结构体，但是如果这个结构体的属性很多就会占据栈中的很多空间，反而会降低效率。一般推荐属性3个以下使用结构体封装
  3. 结构体的赋值为直接赋值，而类的赋值是使用对象的地址

在这里，我们只需要管理存储在堆中的OC对象的回收.其他区域中的数据的回收是系统自动管理的。

野指针与僵尸对象

C语言中的野指针： 定义1个指针变量.没有初始化，其指向一块随机空间

OC中的野指针：即指针所指向的对象已被回收

对象回收的本质：当变量被回收的时候,代表变量占用的字节空间从此以后系统可以分配给别人使用了（但不一定已经分配了），但是字节空间中存储的数据还在.

僵尸对象：已经被释放的对象,但是这个对象所占的空间还没有分配给别人。若其空间已分配给别人的时候系统会报错，反之不会。无法复活1个僵尸对象。打开僵尸对象的实时检查机制，则只要访问了僵尸对象就会报错：

Edit Scheme – Run – Diagnostics – 勾上Zombie Objects

为什么不默认打开僵尸对象检测：一旦打开僵尸对象检测 那么在每访问1个对象的时候 都会先检查这个对象是否为1个僵尸对象,
这样是极其消耗性能的.

避免僵尸对象错误：当1个指针称为野指针以后.将这个指针的值设置nil，当1个指针的值为nil 通过这个指针去调用对象的方法(包括使用点语法)的时候.不会报错. 只是没有任何反应。但是如果通过直接访问属性 -> 就会报错.

引用计数器

每个对象都有一个属性，叫做retainCount，即引用计数器，类型为 unsigned long。 当新创建一个对象时，这个对象的引用计数器的值默认是1。引用计数器用来记录当前有多少人在使用此对象，发送retain信息表+1，release相应-1.当retainCount为0时，系统自动调用dealloc方法回收这个对象。

单个对象的内存管理

内存泄露：指的是1个对象没有被及时的回收.在该回收的时候而没有被回收。一直驻留在内存中,直到程序结束的时候才回收.

单个对象的内存泄露的情况.

1). 有对象的创建,而没有对应的relase

2). retain的次数和relase的次数不匹配.

3). 在不适当的时候,为指针赋值为nil

4). 在方法中为传入的对象进行不适当的retain

如何保证单个对象可以被回收

1). 有对象的创建 就必须要匹配1个relase

2). retain次数和release次数一定要匹配.

3). 只有在指针称为野指针的时候才赋值为nil

4). 在方法中不要随意的为传入的对象retain.

循环引用

当两个对象为循环引用关系的时候，如果两边都使用retain属性就会发生内存泄露的错误（对象在该释放的时候不能被释放）

解决方法：1端使用retain 另外1端使用assign 使用assign的那1端在dealloc中不再需要release了.

自动释放池

原理

存入到自动释放池中的对象,在自动释放池被销毁的时候.会自动调用存储在该自动释放池中的所有对象的release方法.将创建的对象,存入到自动释放池之中. 就不再需要手动的relase这个对象了.

创建

使用

在自动释放池之中调用对象的autorelease方法.就会将这个对象存入到当前自动释放池之中，其中autorelease方法返回的是对象本身

注意

  1. 如果只是将对象的创建代码写在自动释放之中,而没有调用对象的autorelease方法.是不会将这个对象存储到这个自动释放池之中的.
  2. 对象的创建可以在自动释放池的外面。但是autorelease方法的调用一定要放在autoreleasepool {}之中才可以将其放入自动释放池中。
  3. 当自动释放池结束的时候.仅仅是对存储在自动释放池中的对象发送1条创建对象匹配的那个release消息，而不是销毁对象，其他的release也不会受到影响
  4. 在自动释放池当中调用几次autorelease就会将该对象存进自动释放池几次，在自动释放池作用域结束后也就给这个对象发送几次release

ARC内存管理机制

在ARC机制下,retain release dealloc这些方法无法调用.

Automatic Reference Counting，自动引用计数.当ARC开启时, 编译器会自动的在合适的地方插入retain、release、autorelase代码.编译器自动为对象做引用计数. 而作为开发者,完全不需要担心编译器会做错(除非开发者自己错用了ARC).

ARC机制下,对象何时被释放

本质: 对象的引用计数器为0的时候,自动释放.

表象: 只要没有强指针指向这个对象,这个对象就会立即回收.

确认程序是否开启ARC机制.

1). 默认情况下,Xcode开启ARC机制.
2). ARC机制下,不允许调用retain、relase、retainCount、autorelease方法.
3). 在dealloc中 不允许[super dealloc];

ARC下的单个对象的内存管理.

在ARC的机制下: 当1个对象没有任何的强指针指向它的时候 这个对象就会被立即回收.

  1. 当指向对象的所有的强指针被回收的时候,对象就会被立即回收.

  1. 将所有指向对象的强指针赋值为nil的时候.对象就会被立即回收.

强指针与弱指针

强指针：默认情况下,我们声明1个指针就是1个强指针.也可以使用__strong来显示的声明这是1个强指针.例如：

弱指针：使用__weak标识的指针就叫做弱指针.

两种指针的区别

无论是强指针还是弱指针,都是指针,都可以用来存储地址。都可以通过这个指针访问对象的成员。

唯一的区别就是在ARC模式下.他们用来作为回收对象的基准.

如果1个对象没有任何强类型的指针指向这个对象的时候,对象就会被立即自动释放

ARC模式下的对象回收标准

  1. ARC机制下释放1个对象的标准是: 没有任何强指针指向对象的时候,对象就会被释放.如果这个时候有弱指针指向,也会被释放.

  1. 注意新创建的对象不能使用__weak声明，否则刚创建出来就会被释放。

  2. 在ARC机制下. 当对象被回收的时候. 原来指向这个对象的弱指针会被自动设置为nil

ARC机制下的多个对象的内存管理

  1. 什么情况下叫做对象没有强指针向指向.

    指向对象的强指针被回收.

    指向对象的强指针被赋值为nil

  2. 在ARC的机制下,@property参数不能使用retain

    因为retain代表生成的setter方法是MRC的标准的内存管理代码.而我们在ARC的机制下不需要这些代码.

    所以,在ARC机制下的setter方法 什么都不需要做.直接赋值就可以了.

  3. 使用参数strong和weak来控制@property生成的私有属性的强/弱类型（默认为强类型的）

使用建议

1). 在ARC机制下，如果属性的类型是OC对象类型的，绝大多数场景下使用strong。

2). 在ARC机制下，如果属性的类型不是OC对象类型的，使用assign

strong和weak都是应用在属性的类型是OC对象的时候. 属性的类型不是OC对象的时候就使用assign.

3). 即在ARC机制下,将MRC下的retain换位strong

ARC循环引用

在ARC机制下.当两个对象相互引用的时候.如果两边都使用strong 那么就会先内存泄露.

解决方案：1端使用strong 1端使用weak

@property前的修饰词

原子性，与多线程相关的参数

  • atomic(默认)：意味着只有一个线程访问实例变量。这时@property生成的setter方法就会被加上一把线程安全锁（保证正在操作此对象的线程只有一个，若有多个则进行等待再访问）。atomic是线程安全的，至少在当前的访器上是安全的。默认的，但是很少使用。比较慢，这跟ARM平台和内部锁机制有关。

  • nonatomic： 可以被多个线程访问。它的速度比atomic快，但不能保证在多线程环境下的安全性，在单线程和明确只有一个线程访问的情况下广泛使用。

无论在ARC还是在MRC都可以使用.
使用建议: 无论是ARC还是MRC 都使用nonatomic

访问器控制，无论是ARC还是MRC 都可以使用.

  • readwrite(默认):表示同时拥有setter和getter。

  • readonly： 表示只有getter没有setter。

系统默认生成的setter和getter方法的名字为setPropertyName和PropertyName，但是有时候为了语意更明确可能需要自定义访问器的名字，最常见的是BOOL类型，比如标识View是否隐藏的属性hidden。可以这样声明

要注意修改setter或者getter的名字是存在副作用的，可能会使KVC和KVO无法正常工作。无论是ARC还是MRC 都可以使用.

  • assign（默认）：用于值类型，在ARC和MRC的模式下都可以使用。即非OC对象的时候。如int、float、double和NSInteger，CGFloat等表示单纯的复制。还包括不存在所有权关系的对象，比如常见的delegate。

  • strong：只能使用在ARC机制下，是retain的一个可选的替代。表示实例变量对传入的参数要有所有权关系即强引用。strong跟retain的意思相同并产生相同的代码，但是语意上更好更能体现对象的关系。

  • weak： 跟assign的效果相似，不同的是weak在对象被回收之后自动设置为nil。一般连线的UI控件使用weak。weak只能用在iOS 5或以后的版本，对于之前的版本，使用unsafe_unretained。

  • unsafe_unretained：weak的低版本替代。

  • copy：为实例变量保留一个自己的副本。

自动释放池的创建与销毁时间

​ 每一次主线程的消息循环开始的时候会先创建自动释放池，消息循环结束前会释放自动释放池。自动释放池被销毁或耗尽时会向池中所有对象发送release消息，释放池中所有对象。使用NSThread做多线程开发时，需要在线程调度方法中手动添加自动释放池。

什么时候使用自动释放池
  • 创建临时对象的循环时，在循环中创建自动释放池
  • 创建子线程
