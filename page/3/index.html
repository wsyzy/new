基础知识

iOS中的进程和线程

  1. 进程
  • 通常来说一个 App 就是⼀个进程，每个进程之间是独立的，运行在其专用的且受保护的内存空间内。iOS 开发中较少的运用进程间的通信(XPC)，绝⼤部分都使用线程。
  • 通过“活动监视器”可以查看Mac系统中开启的进程
  1. 线程
  • 一个进程有多个线程组成，线程是进程的基本执行单元，一个进程中的所有任务都在线程中执行

iOS中的主线程和非主线程

  • 一个程序运行后默认会开启一个线程，成为“主线程”或“UI线程”，一般用来刷新UI界面，处理UI事件（手势操作等）。几乎所有UIKit提供的类和包含Mutable的类都是线程不安全的，为保证流畅体验及线程安全，所有 UI 操作都集中到一个线程

    主线程一般都用于展示UI，从网络中获取图片为高耗时的操作，若将其置于主线程，该部分长时间的等待会导致整个界面的UI的加载以及滚动的操作十分缓慢，导致卡顿和失真的现象。因此一般将其置于自定义线程中

  • 影响 UI 体验(较⻓时间的操作)尽量放到非主线程

  • 同步 & 异步

    线程同步：多条线程按顺序执行任务

多线程优点

解决程序阻塞问题，提供程序的执行效率

多线程执行原理

单核CPU同一时间，只能处理一个线程，只有一个线程在执行。多线程的同时执行是CPU快速的在多个线程之间的切换。CPU跳读线程的时间足够快就造成了多线程的“同时”执行。

线程的执行都不是受程序员控制的，系统会自动给我们创建辅助线程，线程间切换执行

线程的优先级

调用方法 threadPriority

线程优先级的取值范围为0.0-1.0，默认为0.5

线程的优先级只能保证程序开始执行时分配资源的概率变大或变小，而不能执行绝对的优先顺序

多线程访问共享资源的问题

  1. 共享资源

    一块资源可能会被多个线程共享，也就是多个线程可能会访问同一块资源，比如说同一个对象，同一个变量，同一个文件等。而当这种情况发生后，很容易引发数据错路和数据安全的问题。

  2. 模拟多窗口卖票（设总数10张）

    问题：两窗口可能出现都卖了一张票但是剩余票数都为9的情况，也可能会出现一个窗口先打印剩8张票，另一个窗口后打印剩9张票的情况。即在一个线程执行其数据读取操作的时候另一个线程操作的插入会导致数据的不连续或者整体的偏差

    结论：线程是不安全的，其执行结果很有可能不符合我们的预期

    解决问题：给线程的读取加上安全锁，保证其在读写数据的时候没有其他线程的插入

线程安全锁（互斥锁）

多线程共享资源时

​

​ 任意一个对象内部都有一把互斥锁（变量，默认打开），为了实现线程同步以防止因多线程抢夺资源造成的数据安全问题，但是加锁会影响程序的性能。

​ 加锁：@synchronized (token) {}，其中token为锁对象，执行时会判断其是打开（为1）还是关闭状态（为0）。如果是线程执行到代码块内部会先上锁。如果锁被关闭，再有线程要执行代码块就会先等待直到锁打开才进入。

消息循环

  • 什么是消息循环：

    消息循环就是NSRunloop这个类 ，每个线程都有自己的消息循环。整个事件的驱动基于消息循环。

    主线程的消息循环默认是开启的（需要去检测事件），子线程默认关闭（通常不需要子线程检测事件）。

  • 消息循环的目的：

    类似于死循环，保证程序不退出（与执行完main函数便退出的程序相对应）、负责处理输入事件（输入源和Timer源）、如果没有事件发生则会让程序处于休眠状态。

  • 消息循环之输入事件

    即Runloop接收输入事件来自两种不同的来源，即输入源和定时源。

    1. 输入源

      包括系统接收的鼠标和键盘事件，NSPort 对象和NSConnection对象。

    2. 定时源

      定时器执行的方法中不易执行太耗时的操作，否则会降低用户体验，用户拖拽时会感到卡顿现象。

  • 消息循环的两种运行模式：

    • NSDefaultRunloopModel

      处理除了NSConnection对象以外的输入源

    • NSRunloopCommonModels

      当对象添加进使用此值作为mode的循环时，该对象将被所有声明为common的mode监视

​ 消息循环需要在一定的模式下才能相匹配，当在消息循环中添加了一个定时源时， 消息循环的模式必须和定时源的模式相匹配才能执行：

  • 子线程的消息循环

GCD（Grand Central Dispatch）

函数方法通常以dispatch开头。

  • 高性能的多线程解决方案，⾃动利用 CPU
  • 线程池模式，⾃动分配/调度线程，管理线程的生命周期
  • 通过队列管理多任务, FIFO
  • 对开发者使用队列代替线程的创建 ( 加入到合适的线程 ——> 加⼊入到合适的队列 )

NSOperation

GCD 缺少可定制性和复杂的依赖，NSOperation是相比于GCD更加面向对象的封装，是在Grand Central Dispatch之上构建的API,其中：

  • NSOperationQueue相当于GCD中的队列
  • NSOperation相当于GCD的block中的任务
  • 提供任务之间的依赖关系
  • 支持取消暂停任务，优先级的设置
  • 子类继承实现，符合业务逻辑的定制
  • 队列支持最大并发数设置等

  • Runloop 伴随线程的任务处理循环
  • Runloop 和线程⼀一对应
  • Mode & 状态监听

与delegate的区别

delegate通常只能实现一对一的活动监管，而KVO能够脱离一对一的模式，通过多个对象注册一个事件，让多个对象同时观察这个事件并做出相应反应。

步骤

注册监听

给self.webView添加一个观察者为self，设置监听属性为“estimatedProgress”，当此属性发生变化时（options设置为NSKeyValueObservingOptionNew），self将做出相应反应

哪些属性是可以被监听的呢

– 当该属性注解有key-value时

接收通知

定义监听者self在接收到被监听者通知时作出的反应,其中change对应着注册监听时的options

移除监听

当监听者self被销毁的时候，需要将自己从被监听者self.webView移除掉

其他

  • 该示例监听的为webView加载url的进度情况，可以结合其WkNavigationDelegate中的decidePolicyForNavigationAction（请求，是否加载url）与didFinishNavigation（完成加载）结合使用，观察监听加载的效果

NSURLSession

浏览器有无痕模式和普通模式，两种模式在加载数据请求上是一样的，但策略不一样。普通模式可以看到浏览历史，但是无痕模式不能看到请求历史以及保存不了cookie。系统提供NSURLSession来接收、发送和处理请求，一个session可以创建多个请求（类似于浏览器可以打开多个网页），一个APP可以有多个NSURLSession，NSURLSession可以将NSURLRequest封装为task进行请求接收，控制状态、开始取消、进度等，尽量让多个NSURLRequest用一个NSURLSession。

NSURLSession的组织结构

创建一个session需要相应的配置和策略，创建NSURLRequest使用NSURLSession封装成NSURLSessionTask来进行下载。同时可以使用NSURLSessionDelegate来进行分阶段回调。不同的session策略不同，即传递的configuration不同。如果只是普通加载网络请求，不需要自定义的session，系统提供默认的sharedSession

结构化数据

使用系统函数解析二进制数据文件得到的字典数组（字典组成的数组）语义不明且使用不便，因此可以创建model类以创建实例对应每一个dictionary数组，那么dictionary中的每一个key都会暴露在model类的头文件中，那么我们就可以在model中操作每一个dictionary的属性。大致过程为：NSData–>JSON–>dictionaryArray–>model

实现

根据返回的json文件明确每一个dictionary中有哪些key。对应着这些key创建model中的属性，并定义函数方法来解析这些key中的数据。解析数据后，注意在model中做好容错性判断：返回的类型是否是自己定义的类型（类型转换错误）、是否有此key（属性和对象不一致错误 ）以及该key中是否有内容（即是否为null）
