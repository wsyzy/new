文档基本设置

  • psd文档：Photoshop源文件，保存文档的图像、图层、通道等，可以利用PS再编辑

  • 像素：图像元素，色彩小方格，构成数码影像的基本单位，

  • 分辨率：单位长度中所表达的像素数目，通常以 像素/英寸(pixels per inch，即ppi) 为单位来表示影像分辨率的大小，例如：pixel per inch 每英寸72像素 = 72 ppi

    ppi：数字影像作品使用

    dpi ：dots per inch，即输出的分辨率设置图像输出使用，设置的分辨率

    分辨率越高，对系统的要求越高，占用的内存就会越大，运行的速度也会变慢。
    一般网页浏览的图片分辨率设置为72ppi，印刷作品的ppi不低于300ppi

  • 色彩模式：通常包括两种模式

    rgb颜色模式：即red，green，blue。针对电子元件，有1670万种颜色

    cmyk颜色模式：c为青色，m为洋红色，y为黄色，k为黑色。常用在印刷作品，减少失                                真，但在运行时比较慢

    即使在CMYK模式下工作，PS也必须将CMYK模式转为显示器所使用的RGB模式，对于同样的图像，CMYK需要处理四个通道

    ​ CMYK与RGB模式的转换：用RGB模式来编辑图像，打印前转换为CMYK模式，进行矫正等再输出

  • 背景内容的设置

    ​ ![image-20200628185513194](/Users/qaq/Library/Application Support/typora-user-images/image-20200628185513194.png)

    ​ 常用白色或者黑色，如果需要制作背景色透明的gif小动画则需要将其背景色设置为透明色，最后保存的.gif或者.png格式的时候便是透明的

  • 自定义文档

    在照片类别：

    需要6寸的照片：选择横向4*6英寸 300ppi的文档，或者纵向的 4 * 6英寸 300ppi的文档

    需要7寸的照片：选择横向或者纵向的 5 * 7英寸的300ppi文档

    A4：210*297 mm的300ppi的文档

    在图稿和插图类别：

抠图的基础方法

    缺点：是在原图像上直接擦除不要的部分，会破坏原图像，操作不可逆，一旦操作失误，不可还原。

  • 图层蒙版法
    可逆的抠图方法。蒙，即为遮盖的意思。

图层蒙版的操作方法及原理

蒙版的添加方法

通过图层面板底部的添加图层蒙版按钮添加蒙版，新增的白色图层即为蒙版层，原图层为被蒙版层，蒙版层利用黑 白 灰三种颜色控制被蒙版层图像的显示与隐藏,蒙版层颜色只能在缩略图中看到，画面显示蒙版结果

图层蒙版的工作原理

黑色不显示被蒙版层图像：完全遮盖
灰色半透明显示被蒙版图像：半遮盖
白色显示被蒙版层图像：不遮盖

![image-20200629141016826](/Users/qaq/Library/Application Support/typora-user-images/image-20200629141016826.png)

运用蒙版抠图的优点

  • 利用蒙版层的黑白灰控制蒙版层图像的显隐
  • 操作可逆，只要改变颜色就可以控制图像的显隐，不改变蒙版层的原图像

换发型

填充前景色快捷键：Alt+delete

剪贴蒙板

创建操作

  • 选择上层 -> 右击 -> 创建剪贴蒙板
  • 用图层菜单创建
  • 或者直接选择上层图层，使用快捷Ctrl+Alt+G创建剪贴蒙板）

相关概念

![image-20200629152844305](/Users/qaq/Library/Application Support/typora-user-images/image-20200629152844305.png)

  • 剪贴蒙板：由两个或者两个以上的图层组成，最下面的脚基层，上面的为顶层
  • 顶层：创建剪贴蒙板时选择的图层
  • 基层：它下面的图层

顶层和基层在剪贴蒙板中怎样表达画面颜色和形状

比如在玫瑰花剪影中，剪影比玫瑰花小，即基层完全重叠在顶层中，此时剪贴版显示顶层（即玫瑰）的颜色；彩色纹理和文字只有重叠部分才会显示顶层（即纹理）颜色。

  • 剪贴蒙板的形状始终与基层形状一致，优先显示顶层图像的颜色，当顶层没有颜色时就显示基层颜色。即优先顶层颜色，保持基层形状
  • 剪贴蒙板的优点为不会破坏顶层图像
  • 基层的形状控制顶层图像的显示区域，基层图像以外的区域隐藏，可以改变基层形状从而改变顶层显示
  • 剪贴蒙板为可逆性操作，可以释放剪贴蒙板

iOS中的动画

iOS中动画的实现基于底层的Core Animation，系统基于Core Animation之上提供了更为底层的 UIView动画 和 UIKit组件自带动画

UIView动画

UIView动画是基于对于Core Animation的封装实现的

能实现简单的动画，处理基本的Frame、Alpha和Transform，但是只能定义动画开始和结束时的状态，不能自定义中间过程。

  4. 新建文件定义UIView，自定义View出现的动画

CALayer

注意其常用属性

  • 每个UIView都有一个CALayer负责内容的绘制与动画，在UIView响应UIResponder的时候便执行绘制与动画
  • UIKit的其他组件对应地展示Layer，比如UIScrollView对应有CAScrollLayer，通常不直接对其进行操作，因为CALayer的动画都是隐式的，即是系统默认提供的
  • CALayer有和UIVIew一样的层级结构，即有subLayer

与delegate的区别

delegate通常只能实现一对一的活动监管，而KVO能够脱离一对一的模式，通过多个对象注册一个事件，让多个对象同时观察这个事件并做出相应反应。

步骤

注册监听

给self.webView添加一个观察者为self，设置监听属性为“estimatedProgress”，当此属性发生变化时（options设置为NSKeyValueObservingOptionNew），self将做出相应反应

哪些属性是可以被监听的呢

– 当该属性注解有key-value时

接收通知

定义监听者self在接收到被监听者通知时作出的反应,其中change对应着注册监听时的options

移除监听

当监听者self被销毁的时候，需要将自己从被监听者self.webView移除掉

其他

  • 该示例监听的为webView加载url的进度情况，可以结合其WkNavigationDelegate中的decidePolicyForNavigationAction（请求，是否加载url）与didFinishNavigation（完成加载）结合使用，观察监听加载的效果

  • 使用UIProgressView可以实现进度条的展示，设置改组件的progress值与webView的estimatedProgress相等便可观察到加载进度

  • 系统KVO框架存在 需要移除监听者所可能造成crash 和 方法名及参数过于复杂的问题。对此，我们可以采用常用的开源KVO框架（比如说FaceBook的KVOController，采用block进行回调处理，github获取框架）来解决crash风险以及方法和参数名复杂的问题。

NSURLRequest

NSURLSession

浏览器有无痕模式和普通模式，两种模式在加载数据请求上是一样的，但策略不一样。普通模式可以看到浏览历史，但是无痕模式不能看到请求历史以及保存不了cookie。系统提供NSURLSession来接收、发送和处理请求，一个session可以创建多个请求（类似于浏览器可以打开多个网页），一个APP可以有多个NSURLSession，NSURLSession可以将NSURLRequest封装为task进行请求接收，控制状态、开始取消、进度等，尽量让多个NSURLRequest用一个NSURLSession。

NSURLSession的组织结构

创建一个session需要相应的配置和策略，创建NSURLRequest使用NSURLSession封装成NSURLSessionTask来进行下载。同时可以使用NSURLSessionDelegate来进行分阶段回调。不同的session策略不同，即传递的configuration不同。如果只是普通加载网络请求，不需要自定义的session，系统提供默认的sharedSession

![image-20200630222507240](/Users/qaq/Library/Application Support/typora-user-images/image-20200630222507240.png)

创建完session后，在session中将request封装成task，之后再操作task进行网络的加载，同时使用delegate方法来监听网络加载请求的回调。task根据请求策略分为四种task，他们也为继承的层级结构关系。对于数据加载量较小的需求，可以使用基本的NSURLSessionDataTask

结构化数据

使用系统函数解析二进制数据文件得到的字典数组（字典组成的数组）语义不明且使用不便，因此可以创建model类以创建实例对应每一个dictionary数组，那么dictionary中的每一个key都会暴露在model类的头文件中，那么我们就可以在model中操作每一个dictionary的属性。大致过程为：NSData–>JSON–>dictionaryArray–>model

实现

根据返回的json文件明确每一个dictionary中有哪些key。对应着这些key创建model中的属性，并定义函数方法来解析这些key中的数据。解析数据后，注意在model中做好容错性判断：返回的类型是否是自己定义的类型（类型转换错误）、是否有此key（属性和对象不一致错误 ）以及该key中是否有内容（即是否为null）
