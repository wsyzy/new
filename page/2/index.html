OC与swift混编之–OC中使用swift写的类

前言

写项目时发现有的方法OC时没有，需要用swift来实现的，于是就跳进了混编配置的坑，参照了许多博客，爬了一上午才爬出来，在这总结一下我挖坑和填坑的经历，如有错误，欢迎纠正。

  • 需求：OC项目中使用新建swift文件的类
  • 问题：新建swift文件时，笔者的Xcode没有提示生成桥接文件，需要自己配置并手动创建桥接文件，最后在OC中成功引用swift写的类
  • 项目说明：没有使用framework与pod

重要的配置及文件认识

  1. Target–Building Setting–Packing

    • Defines Module 设置为YES

    目测是配置项目是否接收混编的（目测哈～）

    • Product Module name和Product name设置为项目名

      后面配置参数时需要引用

  2. Target–Building Setting–Swift Compiler-General

    • Objective-C Bridging Header：桥接文件路径配置，桥接文件名必须是“ProjectName-Bridging-Header.h”，这个文件在Swift使用OC类时使用
    • Objective-C Generated Interface Header Name：这个文件路径是混编时，系统自动生成的Swift文件对应的Objective-C的头文件路径，OC中使用Swift时使用。这个文件默认为“工程名-Swift.h”，一般不作改动。

遇到的问题及解决方案

  1. Xcode不能提示自动生成桥接文件

    OC中用Swift时的桥接文件“工程名-Swift.h”：注意，虽然系统没有提示自动生成了该文件，但是是生成了的（划重点），不需要自己再手动创建！！！！（笔者很傻地手动创建了一个，在路径配置和swift类不能识别中浪费了很多时间）。

    Swift中用OC的文件可以自己创建，详情再看看刚刚说的Objective-C Bridging Header

  2. 混编配置过程中Xcode报错

    • 若错误为“XXX……not find” （文件找不到）或者“error opening ……” 一般是文件路径配置错误，注意使用相对路径（直接为创建的头文件名），不要使用绝对路径（/User/……）！！

    • 若错误为“Failed to emit precompiled header for bridging header ”，在buildsetting中的以下两个路径中添加对应的设置，重现编译即可解决

      1. 添加 $(inherited) non-recursive 到 Search Path -> Header Search Paths
      2. 添加 ${PODS_ROOT} recursive 到 Search Path -> User Header Search Paths

学习笔记–iOS中的多线程

Apple视频

基础知识

iOS中的进程和线程

  1. 进程
  • 通常来说一个 App 就是⼀个进程，每个进程之间是独立的，运行在其专用的且受保护的内存空间内。iOS 开发中较少的运用进程间的通信(XPC)，绝⼤部分都使用线程。
  • 通过“活动监视器”可以查看Mac系统中开启的进程
  1. 线程
  • 一个进程有多个线程组成，线程是进程的基本执行单元，一个进程中的所有任务都在线程中执行

iOS中的主线程和非主线程

  • 一个程序运行后默认会开启一个线程，成为“主线程”或“UI线程”，一般用来刷新UI界面，处理UI事件（手势操作等）。几乎所有UIKit提供的类和包含Mutable的类都是线程不安全的，为保证流畅体验及线程安全，所有 UI 操作都集中到一个线程

    主线程一般都用于展示UI，从网络中获取图片为高耗时的操作，若将其置于主线程，该部分长时间的等待会导致整个界面的UI的加载以及滚动的操作十分缓慢，导致卡顿和失真的现象。因此一般将其置于自定义线程中

  • 影响 UI 体验(较⻓时间的操作)尽量放到非主线程

  • 同步 & 异步

    线程同步：多条线程按顺序执行任务

多线程优点

解决程序阻塞问题，提供程序的执行效率

多线程执行原理

单核CPU同一时间，只能处理一个线程，只有一个线程在执行。多线程的同时执行是CPU快速的在多个线程之间的切换。CPU跳读线程的时间足够快就造成了多线程的“同时”执行。

线程的属性

线程执行完之后会自动销毁，当线程死亡之后，不能再次使用

线程的执行都不是受程序员控制的，系统会自动给我们创建辅助线程，线程间切换执行

线程的优先级

调用方法 threadPriority

线程优先级的取值范围为0.0-1.0，默认为0.5

线程的优先级只能保证程序开始执行时分配资源的概率变大或变小，而不能执行绝对的优先顺序

多线程访问共享资源的问题

  1. 共享资源

    一块资源可能会被多个线程共享，也就是多个线程可能会访问同一块资源，比如说同一个对象，同一个变量，同一个文件等。而当这种情况发生后，很容易引发数据错路和数据安全的问题。

  2. 模拟多窗口卖票（设总数10张）

    问题：两窗口可能出现都卖了一张票但是剩余票数都为9的情况，也可能会出现一个窗口先打印剩8张票，另一个窗口后打印剩9张票的情况。即在一个线程执行其数据读取操作的时候另一个线程操作的插入会导致数据的不连续或者整体的偏差

    结论：线程是不安全的，其执行结果很有可能不符合我们的预期

    解决问题：给线程的读取加上安全锁，保证其在读写数据的时候没有其他线程的插入

线程安全锁（互斥锁）

多线程共享资源时

​

​ 任意一个对象内部都有一把互斥锁（变量，默认打开），为了实现线程同步以防止因多线程抢夺资源造成的数据安全问题，但是加锁会影响程序的性能。

​ 加锁：@synchronized (token) {}，其中token为锁对象，执行时会判断其是打开（为1）还是关闭状态（为0）。如果是线程执行到代码块内部会先上锁。如果锁被关闭，再有线程要执行代码块就会先等待直到锁打开才进入。

消息循环

  • 什么是消息循环：

    消息循环就是NSRunloop这个类 ，每个线程都有自己的消息循环。整个事件的驱动基于消息循环。

    主线程的消息循环默认是开启的（需要去检测事件），子线程默认关闭（通常不需要子线程检测事件）。

  • 消息循环的目的：

    类似于死循环，保证程序不退出（与执行完main函数便退出的程序相对应）、负责处理输入事件（输入源和Timer源）、如果没有事件发生则会让程序处于休眠状态。

  • 消息循环之输入事件

    即Runloop接收输入事件来自两种不同的来源，即输入源和定时源。

    1. 输入源

      包括系统接收的鼠标和键盘事件，NSPort 对象和NSConnection对象。

    2. 定时源

      定时器执行的方法中不易执行太耗时的操作，否则会降低用户体验，用户拖拽时会感到卡顿现象。

  • 消息循环的两种运行模式：

    • NSDefaultRunloopModel

      处理除了NSConnection对象以外的输入源

    • NSRunloopCommonModels

      当对象添加进使用此值作为mode的循环时，该对象将被所有声明为common的mode监视

​ 消息循环需要在一定的模式下才能相匹配，当在消息循环中添加了一个定时源时， 消息循环的模式必须和定时源的模式相匹配才能执行：

函数方法通常以dispatch开头。

  • 高性能的多线程解决方案，⾃动利用 CPU

  • 通过队列管理多任务, FIFO
  • 对开发者使用队列代替线程的创建 ( 加入到合适的线程 ——> 加⼊入到合适的队列 )

iOS沙盒机制和文件结构

沙盒机制

大部分情况下，ios中的app只能访问自己沙盒中的文件，而不能对其他app的进行访问。其中，info.plist文件与二进制资源文件存储在Bundle中，而在app开发过程中访问的是文件系统，存储在Datas文件夹中

沙盒文件结构

  • Document：可以进行备份和恢复，体积较大，一般存档用户数据
  • Library：开发者最常使用的文件夹，可以自定义子文件夹
    • preferences：用户偏好设置，NSUserDefault，支持备份
    • Cache：不需要缓存的，体积较大，一般的删除缓存操作
    • 自定义文件夹：Documents，支持用户共享
  • tmp：临时文件不会备份，启动时有可能被清除

原色通道调色

原色通道的亮度值越高，原色色彩饱和度越高，颜色越纯。即若红通道曲线凸起，红通道亮度增加，画面颜色偏红。若此时绿通道曲线向上凸起，画面绿色信息饱和度更高，多个通道的颜色复合计算，得到黄色；若此时红通道曲线向下凹陷，画面呈现青色，即曲线单通道调色时，互补色灰及时响应，从十二色相环中可以知道，红色的互补色为青色（从颜色起点起正旋转180度得到其互补色），
